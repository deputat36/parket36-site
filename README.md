# Паркет36 — мастер Иван / parket36.ru

Статический сайт для заявок на паркетные работы, циклёвку, реставрацию и уход за деревянными полами в Воронеже и Воронежской области.

## Основное позиционирование

- Паркет, циклёвка, реставрация, деревянные полы и укладка покрытий — основная специализация.
- Иван лично принимает обращения и объясняет формат выполнения.
- Смежные бытовые задачи могут обсуждаться только как дополнительные и не должны подменять основное направление сайта.
- Сложная электрика, сантехника, отделка, демонтаж, перевозки и другие профильные задачи передаются отдельным специалистам только после согласования с клиентом.

## Единый источник настроек

Общие настройки вынесены в `data/site.json`:

```json
{
  "brand": "Паркет36",
  "person": "Иван",
  "phone_display": "8 (900) 926-79-29",
  "phone_e164": "+79009267929",
  "domain": "https://parket36.ru",
  "service_area": [
    "Воронеж",
    "Воронежская область"
  ],
  "max_url": "",
  "metrika_id": "",
  "default_request_path": "/zayavka/",
  "current_year": 2026
}
```

От этого файла зависят:

- синхронизация телефона в HTML;
- проверки `tel:`-ссылок;
- проверки основного пути заявки;
- проверка canonical-домена;
- проверка `CNAME`, `robots.txt` и sitemap;
- подключение или удаление блока Яндекс.Метрики;
- сборка публичной папки `_site`.

После изменения телефона, домена, основного пути заявки, персональной ссылки MAX или ID Метрики нужно выполнить:

```bash
python tools/site_settings.py --write
python tools/run_quality_checks.py
```

Пока персональная ссылка MAX не получена, на публичных страницах не должно быть общих MAX-кнопок. Основной канал связи — телефон, а страница `/zayavka/` отправляет заявку через Supabase Edge Function и одновременно копирует готовый текст обращения для отправки фотографий отдельными сообщениями. После заполнения `max_url` инструмент синхронизации заменит уже существующие MAX-ссылки в HTML-страницах, если они будут добавлены вручную.

Поле `metrika_id` пока пустое. Когда появится реальный ID счётчика, нужно указать только цифры, например `"metrika_id": "12345678"`, затем запустить `python tools/site_settings.py --write`. Скрипт сам добавит блок Метрики на HTML-страницы и выставит `window.parket36MetrikaId`, который нужен событиям из `js/main.js`.

## Supabase

Публичная форма оценки по фото использует отдельный контур Паркет36 в проекте Supabase `ofewxuqfjhamgerwzull`:

- Edge Function: `parket-public-lead`;
- таблица заявок: `public.parket_leads`;
- аудит публичных отправок и мягкий антиспам: `public.parket_public_lead_audit`.

Таблицы Паркет36 всегда используют префикс `parket_*`. Не смешивайте их с `leader_*`, `nav_*` или `core_*`: эти пространства относятся к другим продуктам.

Правила безопасности:

- в браузере нет `service_role` или secret key;
- прямой доступ `anon` и `authenticated` к `parket_*` отозван;
- RLS включён, добавлены явные deny-политики для публичных ролей;
- запись выполняется только внутри Edge Function;
- IP для антиспама хранится только как SHA-256 hash в audit-таблице;
- при сбое Supabase форма всё равно копирует текст, чтобы клиент мог отправить обращение Ивану вручную.

Исходник функции хранится в `supabase/functions/parket-public-lead/index.ts`, SQL-схема — в `supabase/sql/parket_leads.sql`.

## Аналитика звонков

Главное бизнес-действие сайта — звонок Ивану. В `js/main.js` уже есть события для кликов по телефону, открытия формы, выбора шаблона задачи, успешной отправки заявки и fallback-копирования текста.

Инструкция по подключению целей Метрики, GTM/dataLayer, ручной проверке событий и UTM-разметке хранится в `docs/analytics-events.md`.

Ключевые события:

- `parket36:phone-click` — отдельное браузерное событие клика по телефону;
- `parket36_phone_click` — событие для `window.dataLayer`;
- `phone-click` — рекомендуемая цель Яндекс.Метрики;
- `request-submit` — успешная заявка через Supabase;
- `request-copy` — fallback, когда текст подготовлен, но автоотправка не прошла.

## Автоматическая проверка

Каждый Pull Request, изменение ветки `main` и еженедельный запуск проверяются workflow `Site quality`.

`Site quality` и `Deploy GitHub Pages` используют один общий quality gate:

```bash
python tools/run_quality_checks.py
```

Внутри него запускаются:

- `site_settings.py --check` — синхронизация общих настроек;
- `check_domain_settings.py` — соответствие `CNAME` и sitemap в `robots.txt` домену из `data/site.json`;
- `check_workflows.py` — версии GitHub Actions и обязательные шаги CI/деплоя;
- `check_quality_runner.py` — точный состав и порядок общего quality gate;
- `check_docs.py` — актуальность README и workflow относительно общего quality runner;
- `check_sitemap_helper.py` — проверка вспомогательного инструмента добавления sitemap-записей;
- `check_site.py` — общий статический аудит страниц;
- `check_guardrails.py` — дополнительные правила фокуса, индексации и внутренних страниц;
- `check_conversion_paths.py` — телефон, мобильная CTA, финальные CTA и ссылки на оценку по фото;
- `check_lead_paths.py` — целостность формы оценки и лидогенерационных путей;
- `build_pages.py` — сборка чистого публичного каталога `_site`.

Проверки покрывают:

- внутренние ссылки и изображения;
- уникальность title и canonical;
- description, Open Graph, theme-color и H1;
- наличие страниц в sitemap;
- соответствие `dateModified` в HTML и `lastmod` в sitemap;
- alt у изображений;
- безопасное подключение основного JavaScript через `defer`;
- версии GitHub Actions и обязательный запуск quality gate перед деплоем;
- фиксированный состав общего quality gate;
- отсутствие WhatsApp, старой цветовой схемы и устаревших ссылок;
- отсутствие видимых SEO-списков;
- корректный `noindex` для 404, политики, страницы-переходника, внутренних рабочих страниц и дополнительных смежных услуг;
- отсутствие ссылок на объединённую страницу `master-na-chas`;
- соответствие контактов файлу `data/site.json`;
- синхронизацию телефонных данных и опционального блока Метрики;
- дополнительные guardrails для фокуса на паркете, публичной страницы оценки по фото и внутренних рабочих страниц;
- конверсионный путь: телефон, мобильная CTA, финальные CTA и ссылки на оценку по фото;
- целостность формы оценки: поля по фото, видео, площади, задаче, времени звонка и контакту;
- успешная сборка чистого публичного каталога;
- отсутствие внутренних рабочих страниц и приватных файлов в публичной сборке.

Локальная проверка перед PR:

```bash
python tools/run_quality_checks.py
```

Ручной отчёт по карточкам раздела советов можно запускать отдельно:

```bash
python tools/report_advice_index_gaps.py
```

## Публикация

Workflow `Deploy GitHub Pages` после изменения `main`:

1. запускает общий quality gate `tools/run_quality_checks.py`;
2. использует собранную папку `_site`;
3. загружает только файлы сайта;
4. публикует артефакт в GitHub Pages.

Для автоматического деплоя в `Settings → Pages` источником должен быть выбран `GitHub Actions`.

## Домен

Файл `CNAME` содержит:

```text
parket36.ru
```

DNS корневого домена должен быть направлен на GitHub Pages. Изменения DNS выполняются один раз в кабинете регистратора домена.

При смене домена нужно обновить оба места:

- `CNAME` — домен без протокола;
- `data/site.json` → `domain` — домен с `https://` и без завершающего слэша.

После этого обязательна полная локальная проверка:

```bash
python tools/run_quality_checks.py
```

## MAX

Персональная ссылка пока не указана. Когда Иван её получит, нужно изменить только поле `max_url` в `data/site.json`, затем запустить синхронизацию.

Пример:

```json
"max_url": "https://max.ru/u/КОД_ИВАНА"
```

## Реальные фотографии и кейсы

В портфолио пока используются тематические иллюстрации. Для каждого нового объекта желательно сохранять:

1. общий вид до начала работ;
2. дефект крупным планом;
3. процесс;
4. общий вид результата;
5. площадь, район, срок и состав работ.

После получения материалов фотографии обрабатываются, оптимизируются и оформляются как отдельный кейс.

Внутренние рабочие материалы по фото и кейсам должны оставаться в репозитории, но не попадать в публичную сборку `_site` и sitemap. Публичные страницы должны вести клиента на `/zayavka/` или на инструкцию `/sovety/kak-sfotografirovat-pol-dlya-ocenki/`, а не на внутренние ТЗ.

Сейчас из публичной сборки исключаются:

- `foto-dlya-sajta/`;
- `portfolio/shablon-kejsa/`.

## Что нельзя добавлять без подтверждения

Не следует публиковать выдуманные цены, отзывы, стаж, гарантию, адрес, сертификаты или фотографии чужих работ. Эти сведения добавляются только после подтверждения Иваном или владельцем проекта.
