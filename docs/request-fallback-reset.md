# Очистка старого текстового fallback

Если браузер запрещает запись в буфер обмена, подробная форма создаёт readonly-textarea с атрибутом `data-request-fallback`. Пользователь может вручную скопировать из неё подготовленный текст.

До исправления textarea оставалась в форме и после новой отправки. Если вторая попытка копирования была успешной, feedback-модуль всё равно видел старый `data-request-fallback` и показывал неверную инструкцию «Скопируйте готовый текст ниже».

## Новое поведение

В capture-обработчике новой валидной отправки `js/lead-notification-feedback.js` выполняет:

1. сброс технического состояния доставки;
2. удаление старого `data-lead-fallback-actions`;
3. удаление старого `data-request-fallback`;
4. запуск текущей попытки через существующий обработчик `main.js`.

Очистка происходит только при новом событии `submit`. Текстовый fallback, созданный текущей попыткой после ошибки clipboard, не удаляется и остаётся доступным пользователю.

После успешного повторного копирования:

- старый textarea отсутствует;
- статус сообщает «Текст скопирован»;
- статус не предлагает копировать текст ниже;
- аварийные действия сохраняются, если уведомление Ивану не подтверждено.

## Проверки

`tests/e2e/request-fallback-reset.spec.mjs` воспроизводит две последовательные отправки:

- первая отклоняет `navigator.clipboard.writeText` и создаёт textarea;
- вторая разрешает clipboard и проверяет удаление textarea и корректный статус.

`tools/check_request_fallback_reset.py` защищает наличие очистки, её порядок перед новой попыткой, браузерный сценарий и эту документацию.

Изменение не читает содержимое textarea, не меняет payload, Supabase, Edge Function или аналитику.
