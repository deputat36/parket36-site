# Проверка телефона в публичной заявке

Форма оценки и короткий callback требуют контакт с пригодным для обратного звонка номером. Одинаковое правило применяется в браузере и в Edge Function.

## Правило

- в поле должно быть от 10 до 15 цифр;
- пробелы, скобки, дефисы, знак `+` и имя клиента допускаются;
- пустое поле по-прежнему обрабатывается стандартным `required` и серверной проверкой обязательных полей;
- короткий или слишком длинный номер блокирует отправку в браузере;
- прямой запрос в Edge Function с непригодным номером получает HTTP 422;
- ошибочное поле получает `aria-invalid`, фокус и понятное сообщение;
- поле получает `autocomplete="tel"` и `inputmode="tel"` для удобного ввода на телефоне.

Примеры допустимого ввода:

- `Иван, +7 (900) 123-45-67`;
- `9001234567`;
- международный номер длиной до 15 цифр.

## Клиентская реализация

`js/contact-validation.js` подключается только к страницам с `#request-form` во время формирования `_site`. Он загружается раньше `lead-reliability.js` и `main.js`, поэтому обычный непригодный контакт блокируется до запроса к backend.

Если сервер дополнительно возвращает `contact_phone_invalid`, скрипт:

- отмечает поле `aria-invalid="true"`;
- возвращает фокус в поле контакта;
- показывает сообщение о необходимости проверить номер;
- не маскирует ошибку общим сообщением о временном сбое.

## Серверная реализация

`supabase/functions/parket-public-lead/contact-validation.ts` содержит чистый валидатор, который считает только цифры. `payload-shape.ts` вызывает его после проверки типа поля и до чтения конфигурации Supabase.

Непригодный непустой контакт возвращает:

```json
{
  "ok": false,
  "error": "contact_phone_invalid",
  "field": "contact",
  "expected": "10-15 digits",
  "received": 5
}
```

HTTP-статус — `422`. Такое отклонение происходит до подключения к базе, rate-limit запросов и создания заявки.

Пустая строка не считается ошибкой формы JSON и передаётся существующей проверке `task_or_contact_required`. Это сохраняет различие между отсутствующим контактом и номером неправильной длины.

## Автоматические проверки

Deno:

```bash
deno test supabase/functions/parket-public-lead/contact-validation_test.ts
deno test supabase/functions/parket-public-lead/payload-shape_test.ts
```

Проверяются:

1. единый диапазон 10–15 цифр;
2. российские и международные форматы с именем и пунктуацией;
3. текст без номера, короткий и чрезмерно длинный номер;
4. ответ `contact_phone_invalid` и количество полученных цифр;
5. сохранение отдельной серверной проверки обязательного контакта.

Playwright в `tests/e2e/contact-validation.spec.mjs` проверяет:

1. короткий номер не вызывает endpoint;
2. обычный российский формат проходит и сохраняется без изменения;
3. серверный ответ `contact_phone_invalid` возвращает фокус и понятное сообщение.

`tools/check_payload_shape.py` защищает серверный модуль, тесты, клиентский fallback и наличие обеих Deno-команд в `Site quality` и `Deploy GitHub Pages`.

## Публикация

Изменение исходника не развёртывает Edge Function автоматически. Production-деплой выполняется только после настройки обязательных secrets и защищённого healthcheck по `docs/lead-endpoint-test-mode.md`.
