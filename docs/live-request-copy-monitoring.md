# Live-контроль текста страницы заявки

Основной сценарий обращения проходит через `/zayavka/`. Поэтому одной проверки главной страницы недостаточно: публичная сборка может обновиться частично или продолжить показывать устаревший текст формы.

## Что проверяется

`tools/check_live_public_copy.py` запрашивает с cache-busting две страницы:

- `/` — клиентская инструкция по фотографиям вместо редакторских заглушек;
- `/zayavka/` — честное разделение сохранения заявки и подтверждения уведомления Ивану.

На странице заявки запрещены безусловные утверждения, что заявка уже ушла, передана или получена Иваном. Обязательны пояснения о попытке сохранения, отдельном статусе уведомления и телефонном резервном действии.

## Когда проверка запускается

Проверка входит в workflow `Live site health`:

- после каждого успешного `Deploy GitHub Pages`;
- ежедневно по расписанию;
- вручную через `workflow_dispatch`.

После deploy выполняется до шести попыток с уникальным query-параметром и заголовками запрета кэширования.

## Результат

Если главная или `/zayavka/` не соответствует контракту, шаг `live_public_copy` завершается ошибкой, диагностический результат попадает в artifact `live-health-report`, а повторный сбой фиксируется monitoring issue.

Проверка не отправляет форму, не создаёт запись в Supabase и не подтверждает работу канала уведомлений. Production Edge Function и контролируемая тестовая заявка проверяются отдельно по задаче #373.

## Локальная проверка контракта

```bash
python tools/check_live_public_copy.py --self-test
python tools/check_live_public_copy_workflow.py
```
