# Privacy-контракт аналитики Паркет36

Аналитика сайта должна измерять действия и техническое состояние воронки, но не копировать персональные данные клиента из формы.

## Что разрешено передавать

В CustomEvent, `dataLayer` и цели Яндекс.Метрики можно передавать только ограниченный контекст:

- технический тип события;
- путь страницы;
- вид формы;
- выбранную категорию услуги;
- безопасные признаки заполнения формы;
- состояние backend и уведомления;
- признак дубликата;
- технический request ID;
- first-touch UTM-атрибуцию;
- безопасный ключ темы callback-формы.

## Что запрещено передавать

Аналитические сигналы не должны содержать:

- имя клиента;
- телефон клиента;
- поле `contact` целиком или частично;
- населённый пункт, улицу или другой текст из поля `location`;
- свободный текст задачи из поля `task`;
- введённое клиентом время связи из `callback_time`;
- готовый текст заявки, который копируется в буфер обмена;
- фотографии, имена файлов и другой пользовательский контент.

Эти данные передаются только в защищённый lead endpoint как содержимое заявки. Сохранение в Supabase и отправка аналитического события являются разными операциями.

## Каналы, которые проверяются

Браузерный тест `tests/e2e/analytics-privacy.spec.mjs` проверяет одновременно:

1. события `parket36:lead`;
2. `parket36:lead-notification`;
3. `parket36:callback-open`;
4. `parket36:callback-request`;
5. `parket36:phone-click`;
6. все записи `window.dataLayer`;
7. все вызовы `window.ym(..., 'reachGoal', ...)`.

Для полной и callback-форм тест подставляет уникальные маркеры в контакт, адрес, задачу и время связи. Затем он подтверждает два независимых факта:

- защищённый lead payload содержит введённые данные и поэтому заявка остаётся полноценной;
- ни один аналитический канал не содержит эти маркеры или номер клиента.

## Fail-closed guardrail

`tools/check_analytics_privacy.py` закрепляет структуру privacy-теста и безопасные поля основных аналитических payload.

Проверка входит в общий quality gate. PR не должен проходить, если:

- privacy-тест удалён или перестал охватывать один из аналитических каналов;
- из теста исчезли уникальные приватные маркеры;
- основное событие успешной заявки начинает включать `contact`, `task`, `locationValue`, `callback_time` или готовый текст;
- notification payload начинает включать контакт, задачу, адрес или время связи;
- документация больше не фиксирует запрет на передачу данных клиента.

## Важное ограничение

Этот контракт не заменяет юридическую проверку политики обработки персональных данных и не подключает реальный счётчик. Он только гарантирует, что текущий клиентский код не копирует содержимое форм в аналитические события.

После получения `metrika_id` необходимо отдельно:

1. подключить счётчик;
2. создать цели;
3. выполнить тестовый визит с UTM;
4. проверить события в интерфейсе Метрики;
5. убедиться, что фактические параметры целей соответствуют этому privacy-контракту.

## Локальная проверка

```bash
python tools/check_analytics_privacy.py
npm run build:site
npm run test:e2e -- tests/e2e/analytics-privacy.spec.mjs
python tools/run_quality_checks.py
```
