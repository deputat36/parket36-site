# Фокус аварийных действий заявки

После сохранения заявки автоматическое уведомление Ивану может быть не подтверждено. В этом случае рядом со статусом появляется блок `data-lead-fallback-actions` с прямым звонком, а в callback-форме — также с переходом к оценке по фото.

## Клавиатурный сценарий

Если после отправки фокус остаётся:

- на кнопке отправки;
- на статусе формы;
- на самой форме;
- либо на `body`,

блок аварийных действий получает программный фокус. Он имеет `tabindex="-1"`, поэтому не создаёт лишнюю постоянную остановку при обычном обходе страницы. Следующее нажатие Tab переводит пользователя на кнопку «Позвонить Ивану».

Блок связан со статусом через `aria-describedby="request-status"`. Экранный диктор получает фактическое предупреждение вместе с назначением группы «Быстрая связь с Иваном».

## Когда фокус не переносится

Интерфейс не должен мешать пользователю, если он уже выбрал другой элемент. В частности, фокус сохраняется:

- на textarea `data-request-fallback`, созданном при недоступном clipboard;
- на любом другом поле или элементе, который пользователь самостоятельно выбрал во время ожидания ответа.

Успешное состояние `notification: sent` не создаёт аварийный блок и не меняет фокус.

## Технические границы

Фокус-логика находится в `js/lead-notification-feedback.js` и не:

- вызывает `fetch`;
- читает контакт, телефон или описание задачи;
- меняет payload заявки;
- изменяет состояние доставки;
- создаёт дополнительные аналитические события.

## Проверка

`tests/e2e/lead-fallback-focus.spec.mjs` проверяет:

1. перевод фокуса на аварийную группу при `notification: disabled`;
2. связь группы со статусом через `aria-describedby`;
3. переход по Tab к ссылке «Позвонить Ивану»;
4. сохранение фокуса на ручном textarea при ошибке clipboard.

Статический контракт закреплён в `tools/check_lead_notification_feedback.py`.
