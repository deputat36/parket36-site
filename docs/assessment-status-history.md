# История статуса подробной заявки

Страница `/zayavka/` использует live-region `#request-status`, чтобы сообщать посетителю о ходе и результате отправки.

## Проблема

Backend может сохранить заявку, но вернуть состояние уведомления:

- `disabled`;
- `partial_failure`;
- `unknown`.

В этих случаях нельзя даже кратковременно показывать текст «Заявка отправлена Ивану». Такой текст может попасть в визуальный кадр или быть озвучен экранным диктором до того, как другой модуль заменит его предупреждением.

## Защита

`js/request-status-tone.js` устанавливает синхронный guard только для подробной формы:

- callback-форма исключена через `data-form-kind="callback"`;
- модуль читает только `window.parket36LastLeadDelivery`;
- подтверждение успеха пропускается без изменений только при `notification: sent`;
- при `disabled`, `partial_failure` или `unknown` оптимистичный текст заменяется до записи в DOM;
- при недоступном clipboard учитывается наличие `[data-request-fallback]`;
- контакт, телефон, описание задачи и payload не читаются.

Модуль не вызывает `fetch`, не отправляет аналитику и не меняет backend-контракт.

## Тексты

При подтверждённом уведомлении подробная форма сохраняет обычное сообщение:

```text
Заявка отправлена Ивану. Текст также скопирован — приложите к нему фотографии пола.
```

При неподтверждённом уведомлении первое итоговое сообщение сразу объясняет фактическое состояние и предлагает позвонить Ивану.

Если clipboard недоступен, первое итоговое сообщение сразу предлагает скопировать textarea ниже. Ложного промежуточного подтверждения быть не должно.

## Проверка

`tests/e2e/assessment-status-history.spec.mjs` записывает каждое непустое изменение `#request-status` через `MutationObserver` и проверяет:

1. `notification: disabled` при рабочем clipboard;
2. `notification: partial_failure` при ручном textarea-fallback;
3. отсутствие любого сообщения, начинающегося с «Заявка отправлена Ивану»;
4. итоговый `data-status-tone="warning"`;
5. наличие правильного действия для посетителя.

Статический контракт `tools/check_request_status_tones.py` дополнительно проверяет синхронный guard, исключение callback-формы, read-only использование delivery-состояния и наличие обоих E2E-сценариев.
