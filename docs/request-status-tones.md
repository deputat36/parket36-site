# Тоны статуса формы заявки

Статус формы должен показывать не только текст, но и смысл результата. Ошибка, ожидание, предупреждение и подтверждённый успех не должны выглядеть одинаково.

## Состояния

Клиентский модуль `js/request-status-tone.js` назначает элементу `#request-status` атрибут `data-status-tone`.

### `info`

Нейтральное информационное сообщение:

- заявка отправляется;
- шаблон задачи добавлен;
- пользователь должен уточнить или проверить данные.

Оформление использует спокойный синий фон и не обещает успешную отправку.

### `success`

Подтверждённый успешный сценарий:

- backend сохранил заявку;
- ответ содержит `notification: sent`.

Оформление зелёное. Сам цвет не заменяет текст и не означает, что Иван уже прочитал уведомление.

### `warning`

Заявка или номер сохранены, но уведомление Ивану не подтверждено:

- `notification: disabled`;
- `notification: partial_failure`;
- `notification: unknown`;
- повторная заявка с `duplicate: true`.

Оформление жёлтое. Рядом остаются аварийные действия: звонок Ивану и, для callback-формы, переход к оценке по фото.

### `error`

Действие требует исправления или ручного продолжения:

- не заполнено обязательное поле;
- телефон не прошёл проверку;
- backend не принял заявку;
- автоматическая отправка не удалась и используется текстовый fallback.

Оформление красное. Форма сохраняет доступный ручной путь и не утверждает, что заявка сохранена.

## Доступность

Для `#request-status` устанавливаются:

```text
role="status"
aria-live="polite"
aria-atomic="true"
```

Изменения сообщаются вспомогательным технологиям целиком, но не прерывают пользователя агрессивным alert-режимом.

Цвет является дополнительным сигналом. В каждом состоянии остаётся самостоятельный понятный текст.

## Подключение

`tools/build_pages.py` добавляет:

```html
<script src="/js/request-status-tone.js" defer></script>
```

только в публичные страницы, содержащие `id="request-form"`.

Порядок скриптов:

1. contact validation и lead reliability;
2. `main.js`;
3. `lead-notification-feedback.js`;
4. `request-status-tone.js`.

Так модуль получает итоговый текст и уже обогащённое событие заявки, но не вмешивается в отправку.

## Границы

Модуль:

- не вызывает `fetch`;
- не читает контакт, текст задачи или другие поля формы;
- не меняет payload;
- не пишет в `dataLayer` и Яндекс Метрику;
- не обращается к Supabase;
- не меняет бизнес-логику сохранения или уведомлений.

Он отвечает только за доступное визуальное представление уже существующего результата.

## Проверки

`tests/e2e/request-status-tone.spec.mjs` проверяет:

1. информационный статус после выбора шаблона;
2. ошибку телефона;
3. подтверждённый `notification: sent`;
4. предупреждение при `notification: disabled`;
5. ошибку backend с сохранённым аварийным действием;
6. ARIA-атрибуты статусного сообщения.

`tools/check_request_status_tones.py` защищает:

- четыре разрешённых тона;
- соответствующие CSS-селекторы;
- отсутствие доступа к персональным данным и сетевым вызовам;
- подключение только к страницам с формой;
- безопасный порядок после lead notification feedback;
- наличие E2E и документации;
- обязательное включение проверки в общий quality gate.
