# История статуса формы обратного звонка

Статус короткой формы на `/kontakty/` является `aria-live="polite"`. Поэтому важен не только итоговый текст, но и каждое промежуточное сообщение: экранный диктор может озвучить текст, который обычный посетитель почти не успеет заметить.

## Проблема

Edge Function отдельно сообщает состояние автоматического уведомления Ивану:

- `sent` — вызов настроенного канала подтверждён;
- `disabled` — канал уведомлений не настроен;
- `partial_failure` — настроенный канал не подтвердил доставку;
- `unknown` — поле отсутствует или имеет неизвестное значение;
- `duplicate: true` без `notification` — заявка уже была сохранена, но повторное уведомление не отправлялось.

Раньше `callback-form.js` при любом `request-submit` сначала устанавливал текст:

```text
Заявка на обратный звонок отправлена Ивану. Он свяжется по указанному номеру.
```

Затем `lead-notification-feedback.js` заменял его предупреждением, если `notification` отличался от `sent`. Итоговый экран был честным, но live-region мог успеть озвучить первое ложное обещание.

## Текущее поведение

`callback-form.js` читает фактическое состояние доставки до первой записи итогового текста.

### `notification: sent`

Показывается обычное подтверждение:

```text
Заявка на обратный звонок отправлена Ивану. Он свяжется по указанному номеру.
```

### `disabled`, `partial_failure`, `unknown`

Первым итоговым сообщением сразу становится предупреждение: номер сохранён, но автоматическое уведомление Ивану не подтверждено. Посетителю предлагается позвонить по номеру `8 (900) 926-79-29`.

### `duplicate: true`

Форма сообщает, что номер уже был сохранён, а повторная отправка не подтверждает новое автоматическое уведомление Ивану.

`lead-notification-feedback.js` по-прежнему отвечает за общий единообразный текст, аварийные кнопки и аналитику. Изменение в callback-модуле устраняет только промежуточное ложное состояние до его обработки.

## Границы

Изменение:

- не меняет payload заявки;
- не создаёт повторное уведомление при duplicate;
- не обращается к Supabase дополнительно;
- не меняет события аналитики;
- не обещает фактическое прочтение сообщения Иваном.

## Проверка

`tests/e2e/callback-status-history.spec.mjs` подключает `MutationObserver` к `#request-status` и сохраняет каждое непустое сообщение.

Проверяется:

1. при `notification: disabled` история не содержит фразу `Он свяжется по указанному номеру`;
2. история содержит предупреждение о ненастроенном уведомлении;
3. итоговый тон равен `warning`;
4. при `notification: sent` обычное подтверждение сохраняется и получает тон `success`.

`tools/check_callback_status_history.py` защищает:

- чтение delivery до установки callback-статуса;
- наличие delivery-aware helper;
- отсутствие старого безусловного присваивания;
- E2E-проверку полной истории статуса;
- соответствие документации и общего quality gate.
