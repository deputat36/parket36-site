# Уведомления о новых заявках Паркет36

Исходник: `supabase/functions/parket-public-lead/index.ts`.

Уведомления отправляются только после успешного сохранения заявки в `parket_leads`. Telegram и email можно включать независимо или одновременно.

## Поддерживаемые каналы

### Telegram

Необходимые secrets Edge Function:

- `PARKET_TELEGRAM_BOT_TOKEN` — токен Telegram-бота;
- `PARKET_TELEGRAM_CHAT_ID` — ID личного чата, группы или канала, куда нужно отправлять заявку.

Оба значения обязательны. Если указан только один параметр, healthcheck вернёт ошибку частичной конфигурации.

### Email через Resend

Необходимые secrets:

- `PARKET_RESEND_API_KEY` — API-ключ Resend;
- `PARKET_EMAIL_FROM` — подтверждённый отправитель в формате `Имя <email@domain.ru>`;
- `PARKET_EMAIL_TO` — один адрес или несколько адресов через запятую;
- `PARKET_EMAIL_SUBJECT` — необязательная тема письма.

Если тема не указана, используется `Новая заявка Паркет36`.

## Что входит в уведомление

- ID заявки;
- контакт клиента;
- выбранная услуга;
- город или район;
- площадь;
- удобное время звонка;
- описание задачи;
- страница, с которой отправлена форма.

Токены, соль IP-хэша, service role key и другие секреты в сообщение не включаются.

## Порядок обработки

1. Edge Function проверяет origin, размер запроса, honeypot и rate limit.
2. Заявка сохраняется в `parket_leads`.
3. Настроенные каналы уведомления запускаются параллельно.
4. Каждый внешний запрос ограничен четырьмя секундами.
5. Результат записывается в `parket_public_lead_audit`.
6. Клиент получает успешный ответ, даже если Telegram или email временно недоступны.

Такой порядок исключает потерю заявки из-за сбоя внешнего сервиса уведомлений.

## Audit-статусы

Для принятой заявки используются причины:

- `accepted_notified` — все настроенные уведомления отправлены;
- `accepted_notification_disabled` — каналы не настроены;
- `accepted_notification_failed` — хотя бы один настроенный канал завершился ошибкой.

В `payload_summary.notification` сохраняются только канал, факт настройки и результат. Текст ошибки записывается в серверный лог `parket_public_lead_notification_failed`, но не возвращается клиенту.

## Проверка конфигурации

Защищённый `test_mode` из `docs/lead-endpoint-test-mode.md` дополнительно показывает:

- `telegram_notification`;
- `email_notification`.

Возможные состояния:

- `disabled` — канал полностью выключен;
- `configured` — все обязательные параметры присутствуют;
- `partially configured` — указан только часть параметров, healthcheck возвращает HTTP 503.

Тестовый режим не отправляет Telegram-сообщение или письмо. Он только проверяет полноту конфигурации.

## Рекомендуемый порядок включения

1. Развернуть актуальную Edge Function.
2. Настроить `PARKET_IP_HASH_SALT` и `PARKET_HEALTHCHECK_TOKEN`.
3. Добавить secrets выбранного канала.
4. Выполнить защищённый healthcheck.
5. Отправить одну реальную тестовую заявку.
6. Проверить строку в `parket_leads`.
7. Проверить audit-причину и фактическое получение уведомления.

## Важные ограничения

- Изменение файла в GitHub не развёртывает Edge Function автоматически.
- Реальные secrets нельзя добавлять в репозиторий, HTML, JavaScript, issue или workflow-логи.
- Для email отправитель должен быть разрешён в аккаунте Resend.
- Telegram-бот должен иметь право писать в выбранный чат.
- Уведомление не заменяет хранение заявки в базе: источником данных остаётся `parket_leads`.
